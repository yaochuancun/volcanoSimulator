# volcanoSimulator 深度技术分析

## 概述

volcanoSimulator 是一个基于 Go 语言实现的 Volcano 调度器仿真框架，专门用于模拟 Kubernetes 上的批处理工作负载调度。该仿真器通过 HTTP API 与 Python 客户端交互，支持配置驱动的集群仿真、作业调度以及 FlexNPU 资源管理。

## 1. 核心调度算法分析

### 1.1 Gang Scheduling（ gang 调度插件）

**源码位置**: `pkg/scheduler/plugins/gang/gang.go`

#### 核心机制

Gang 调度确保属于同一作业（Job）的所有任务（Task）能够**"全有或全无"**地被调度。这是分布式训练作业的关键需求。

```go
// 关键验证逻辑：检查作业是否满足 Gang 调度条件
func (gp *gangPlugin) OnSessionOpen(ssn *framework.Session) {
    validJobFn := func(obj interface{}) *api.ValidateResult {
        job, ok := obj.(*api.JobInfo)
        
        // 1. 检查每个任务类型的最小可用性
        if valid := job.CheckTaskMinAvailable(); !valid {
            return &api.ValidateResult{
                Pass:    false,
                Reason:  v1beta1.NotEnoughPodsOfTaskReason,
                Message: "Not enough valid pods of each task for gang-scheduling",
            }
        }

        // 2. 检查整体任务数量是否满足 minAvailable
        vtn := job.ValidTaskNum()
        if vtn < job.MinAvailable {
            return &api.ValidateResult{
                Pass:   false,
                Reason: v1beta1.NotEnoughPodsReason,
                Message: fmt.Sprintf("Not enough valid tasks for gang-scheduling, valid: %d, min: %d",
                    vtn, job.MinAvailable),
            }
        }
        return nil
    }
    ssn.AddJobValidFn(gp.Name(), validJobFn)
}
```

#### 作业就绪判断

```go
// Ready() 方法判断作业是否就绪
func (ji *JobInfo) Ready() bool {
    occupied := ji.ReadyTaskNum()
    return occupied >= ji.MinAvailable
}

// ReadyTaskNum 计算已就绪的任务数
func (ji *JobInfo) ReadyTaskNum() int32 {
    occupied := 0
    occupied += len(ji.TaskStatusIndex[Bound])
    occupied += len(ji.TaskStatusIndex[Binding])
    occupied += len(ji.TaskStatusIndex[Running])
    occupied += len(ji.TaskStatusIndex[Allocated])
    occupied += len(ji.TaskStatusIndex[Succeeded])
    // ...
    return int32(occupied)
}
```

#### Gang 调度的抢占保护

```go
preemptableFn := func(preemptor *api.TaskInfo, preemptees []*api.TaskInfo) ([]*api.TaskInfo, int) {
    var victims []*api.TaskInfo
    jobOccupiedMap := map[api.JobID]int32{}

    for _, preemptee := range preemptees {
        job := ssn.Jobs[preemptee.Job]
        if _, found := jobOccupiedMap[job.UID]; !found {
            jobOccupiedMap[job.UID] = job.ReadyTaskNum()
        }

        // 关键：只有当作业的就绪任务数 > MinAvailable 时才允许被抢占
        if jobOccupiedMap[job.UID] > job.MinAvailable {
            jobOccupiedMap[job.UID]--
            victims = append(victims, preemptee)
        }
    }
    return victims, util.Permit
}
```

#### 算法特点

| 特性 | 说明 |
|------|------|
| **MinAvailable** | 作业启动所需的最小任务数 |
| **TaskMinAvailable** | 每种任务类型的最小可用数 |
| **Pipelined 状态** | 支持资源预分配（资源将释放时） |
| **抢占保护** | 保护已就绪的 Gang 作业不被抢占 |

### 1.2 DRF (Dominant Resource Fairness)

**源码位置**: `pkg/scheduler/plugins/drf/drf.go`

#### 核心概念

DRF 是一种**多资源公平分配算法**，确保每个作业获得其主导资源（使用占比最高的资源）的公平份额。

```go
type drfAttr struct {
    share            float64    // 主导资源份额
    dominantResource string     // 主导资源名称
    allocated        *api.Resource  // 已分配资源
}
```

#### 份额计算

```go
func (drf *drfPlugin) calculateShare(allocated, totalResource *api.Resource) (string, float64) {
    res := float64(0)
    dominantResource := ""
    
    // 遍历所有资源类型，找出使用占比最高的（主导资源）
    for _, rn := range totalResource.ResourceNames() {
        share := helpers.Share(allocated.Get(rn), totalResource.Get(rn))
        if share > res {
            res = share
            dominantResource = string(rn)
        }
    }
    return dominantResource, res
}
```

#### 作业排序

```go
jobOrderFn := func(l interface{}, r interface{}) int {
    lv := l.(*api.JobInfo)
    rv := r.(*api.JobInfo)

    // 份额小的作业优先（更"饥饿"的作业先调度）
    if drf.jobAttrs[lv.UID].share == drf.jobAttrs[rv.UID].share {
        return 0
    }
    if drf.jobAttrs[lv.UID].share < drf.jobAttrs[rv.UID].share {
        return -1
    }
    return 1
}
```

#### Namespace 公平性

```go

    // 计算加权份额：share / weight
    lWeightedShare := lOpt.share / float64(lWeight)
```

### 1.3 Binpack 调度

**源码位置**: `pkg/scheduler/plugins/binpack/binpack.go`

#### 核心思想

Binpack 采用**最佳适应（Best Fit）**策略，优先将任务调度到资源利用率高的节点，减少资源碎片化。

```go
// BinPackingScore 计算节点评分
func BinPackingScore(task *api.TaskInfo, node *api.NodeInfo, weight priorityWeight) float64 {
    score := 0.0
    weightSum := 0
    
    for _, resource := range requested.ResourceNames() {
        request := requested.Get(resource)
        allocate := allocatable.Get(resource)
        nodeUsed := used.Get(resource)

        // 计算单个资源的 binpack 分数
        resourceScore := ResourceBinPackingScore(request, allocate, nodeUsed, resourceWeight)
        score += resourceScore
        weightSum += resourceWeight
    }

    // 归一化到 [0, 10]
    if weightSum > 0 {
        score /= float64(weightSum)
    }
    score *= float64(k8sFramework.MaxNodeScore * int64(weight.BinPackingWeight))
    return score
}
```

#### 资源评分公式

```go
func ResourceBinPackingScore(requested, capacity, used float64, weight int) float64 {
    if capacity == 0 || weight == 0 {
        return 0
    }

    usedFinally := requested + used
    if usedFinally > capacity {
        return 0
    }

    // 分数 = 使用后资源占比 * 权重
    // 使用越多，分数越高（Best Fit）
    score := usedFinally * float64(weight) / capacity
    return score
}
```

#### 可配置权重

```yaml
plugins:
  - name: binpack
    arguments:
      binpack.weight: 10      # 整体权重
      binpack.cpu: 5          # CPU 权重
      binpack.memory: 1       # 内存权重
      binpack.resources: nvidia.com/gpu
      binpack.resources.nvidia.com/gpu: 2
```

### 1.4 调度 Action 流水线

**源码位置**: `pkg/scheduler/actions/`

Volcano 采用**Action 流水线**架构，调度周期按顺序执行多个 Action：

```
典型流水线: enqueue → reclaim → allocate → backfill → preempt
```

#### Allocate Action（核心分配）

```go
func (alloc *Action) Execute(ssn *framework.Session) {
    // 1. 构建 Namespace → Queue → Job 的层次结构
    jobsMap := map[api.NamespaceName]map[api.QueueID]*util.PriorityQueue{}
    
    // 2. 按 Namespace → Queue → Job 的顺序选择
    for !namespaces.Empty() {
        namespace := namespaces.Pop().(api.NamespaceName)
        queue := selectBestQueue(jobsMap[namespace])
        job := jobs.Pop().(*api.JobInfo)
        
        // 3. 为作业的 Pending 任务分配资源
        for _, task := range job.TaskStatusIndex[api.Pending] {
            // 过滤节点
            predicateNodes, fitErrors := ph.PredicateNodes(task, nodes, predicateFn)
            
            // 选择最佳节点
            nodeScores := util.PrioritizeNodes(task, candidateNodes, ...)
            node := util.SelectBestNode(nodeScores)
            
            // 分配或流水线
            if task.InitResreq.LessEqual(node.Idle, api.Zero) {
                stmt.AllocateV2(task, node)
            } else if task.InitResreq.LessEqual(node.FutureIdle(), api.Zero) {
                stmt.Pipeline(task, node.Name)
            }
        }
        
        // 4. 提交或回滚
        if ssn.JobReady(job) {
            stmt.Commit()
        } else if !ssn.JobPipelined(job) {
            stmt.Discard()
        }
    }
}
```

#### Enqueue Action（入队）

```go
func (enqueue *Action) Execute(ssn *framework.Session) {
    // 将 Pending 状态的 Job 按队列排序后入队
    for _, job := range ssn.Jobs {
        if job.IsPending() {
            if job.PodGroup.Spec.MinResources == nil || ssn.JobEnqueueable(job) {
                ssn.JobEnqueued(job)
                job.PodGroup.Status.Phase = scheduling.PodGroupInqueue
            }
        }
    }
}
```

#### Preempt Action（抢占）

```go
func (pmpt *Action) Execute(ssn *framework.Session) {
    // 1. 识别"饥饿"作业（ starving jobs）
    for _, job := range ssn.Jobs {
        if ssn.JobStarving(job) {
            preemptorsMap[job.Queue].Push(job)
        }
    }
    
    // 2. 对饥饿作业的待调度任务进行抢占
    for _, queue := range queues {
        for !preemptors.Empty() {
            job := preemptors.Pop().(*api.JobInfo)
            task := preemptorTasks[job.UID].Pop().(*api.TaskInfo)
            
            // 寻找可抢占的受害者
            preempted, _ := preempt(ssn, stmt, preemptor, filter, ph)
            
            if ssn.JobPipelined(job) {
                stmt.Commit()
            } else {
                stmt.Discard()
            }
        }
    }
}
```

### 2.2 资源粒度控制

**源码位置**: `input_config/input_config_loader.py`

```python
_FLEXNPU_CORE_KEY = "volcano.sh/flexnpu-core.percentage"

def _ceil_to_step(value: float, step: float) -> float:
    """将 value 向上取整到 step 的整数倍"""
    if step <= 0:
        return value
    return math.ceil(value / step) * step

def _round_resource_map(resources: Optional[Dict[str, Any]], granularity_percent: float) -> None:
    """按粒度向上取整 flexnpu_core 请求"""
    if not resources or granularity_percent <= 0:
        return
    key = _FLEXNPU_CORE_KEY
    if key not in resources:
        return
    raw = resources[key]
    v = float(raw)
    rounded = _ceil_to_step(v, float(granularity_percent))
    resources[key] = str(int(rounded))
```

**示例**:
```yaml
spec:
  npuGranularityPercent: 25  # 25% 粒度

# 请求 50% 算力 → 实际分配 50%（已是 25 的倍数）
# 请求 30% 算力 → 实际分配 50%（向上取整）
# 请求 10% 算力 → 实际分配 25%（向上取整）
```

## 4. 性能特性与瓶颈分析

### 4.1 时间复杂度分析

| 操作 | 时间复杂度 | 说明 |
|-----|-----------|------|
| 打开 Session | O(J + N) | J=作业数, N=节点数 |
| 节点过滤 (Predicate) | O(N × T) | T=每个作业的任务数 |
| 节点评分 (Prioritize) | O(N × T) | 对每个候选节点评分 |
| 作业排序 | O(J log J) | 使用优先队列 |
| 队列排序 | O(Q × J) | Q=队列数 |
| Allocate Action | O(J × T × N × P) | P=谓词计算开销 |

## 6. 总结

### 6.1 架构亮点

1. **清晰的层次结构**: Session → Action → Plugin 三层架构，职责明确
2. **事务支持**: Statement 机制支持调度决策的原子提交/回滚
3. **插件化设计**: 丰富的扩展点支持自定义调度策略
4. **仿真能力**: 完整的仿真框架支持离线实验和策略验证

### 6.2 FlexNPU 创新点

1. **细粒度资源划分**: 支持百分比级别的算力分配
2. **粒度控制**: 可配置的分配粒度（如 25% 步长）
3. **多卡支持**: Pod 可以指定需要多张卡（flexnpu-num）
4. **拓扑感知**: 支持 HCCS、SIO 等 NPU 拓扑

1. **性能优化**:
