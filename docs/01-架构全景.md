# 01-架构全景：系统是如何组织的？

> 本文档从架构视角拆解 VolcanoSimulator，帮助你理解系统为什么这样设计，以及各个模块之间如何协作。

---

## 一、架构设计哲学

### 三个核心原则

```
┌────────────────────────────────────────────────────────────────┐
│  1. 真实代码仿真                                                │
│     └── 运行真实的 Volcano 调度代码，不是简化版                 │
│                                                                │
│  2. 配置驱动                                                    │
│     └── 用 YAML 描述实验场景，无需修改代码即可切换配置          │
│                                                                │
│  3. 前后端分离                                                  │
│     └── Python 负责编排和报表，Go 负责核心调度，HTTP 解耦       │
└────────────────────────────────────────────────────────────────┘
```

### 为什么要这样设计？

**为什么不全部用 Python 写？**
- Volcano 是用 Go 写的，全部重写工作量巨大
- 用真实代码才能保证仿真结果有参考价值

**为什么不全部用 Go 写？**
- Python 在数据处理、报表生成方面更便捷
- YAML 解析和配置管理用 Python 更灵活

**为什么要 HTTP 通信而不是直接调用？**
- 解耦：两边可以独立开发、独立重启
- 可扩展：未来可以支持远程仿真器、多客户端等
- 可调试：可以用 curl/postman 直接测试仿真器接口

## 三、模块详解

### 3.1 Python 应用层

#### SimRun.py（指挥员）

**职责：** 整个仿真流程的 orchestrator（编排者）

**工作流程：**
```
① 读配置 → ② 初始化(reset) → ③ 调度(step) → ④ 获取结果 → ⑤ 生成报表
     ↑___________________________________________________________↓
                        （循环直到完成）
```

**关键设计：**
- 与 Go 仿真器是 **"请求-响应"** 模式，不是长连接
- 每次 step 后需要轮询 stepResult，因为调度是异步完成的

#### input_config（配置系统）

**三个核心文件：**

| 文件 | 职责 | 类比 |
|------|------|------|
| `input_config_loader.py` | 把用户写的 YAML 转成仿真器能理解的格式 | 翻译官 |
| `flexnpu_util_report.py` | 分析 GPU 利用率，生成报告 | 分析师 |
| `output_csv_reports.py` | 把仿真结果写成 CSV 文件 | 文书 |

**设计亮点：**
- **数据转换分离**：加载器负责格式转换，不碰业务逻辑
- **双轨统计**：区分「原始需求」和「实际分配」

---

### 3.2 Go 核心层

#### HTTP Server（接口网关）

**四个接口的设计意图：**

| 接口 | 作用 | 类比 |
|------|------|------|
| `POST /reset` | 初始化/重置整个仿真环境 | 游戏重新开始 |
| `POST /step` | 推进一轮调度 | 游戏下一回合 |
| `GET /stepResult` | 获取当前状态（可能未就绪） | 问"好了吗？" |
| `GET /stepResultAnyway` | 强制获取当前状态 | 不管好没好都要看 |

**为什么 stepResult 可能返回 "0"？**

因为调度是 **异步** 的：
1. Python 发送 /step 请求（瞬间完成）
2. Go 收到请求，标记 "需要执行调度"
3. Go 的主循环在下一秒执行调度
4. 调度完成后才能提供有效结果

所以 stepResult 在调度完成前返回 "0"，完成后返回 JSON 数据。

#### 主仿真循环（时间推进器）

**伪代码：**
```go
for true {
    // 1. 检查是否有新 Job 到提交时间
    submitJobsThatAreDue()
    
    // 2. 推进容器创建（Binding → Running）
    advanceContainerCreation()
    
    // 3. 等待新的调度配置（如果到了配置周期）
    waitForNewSchedulerConf()
    
    // 4. 执行调度
    session := OpenSession(cluster, tiers, cfg)
    for _, action := range actions {
        action.Execute(session)
    }
    
    // 5. 同步 Pod 状态
    syncPodPhases()
    
    // 6. 时间 +1 秒
    NowTime += 1 second
}
```

**关键设计：**
- **离散时间**：时间以 1 秒为粒度推进（不是连续的）
- **事件驱动**：Job 提交、容器创建、调度执行都是事件
- **状态机**：Pod 从 Pending → Binding → Running 的状态转换

#### Volcano 调度框架（调度大脑）

**三层结构：**

```
┌─────────────────────────────────────────┐
│           Action 层（动作）              │
│  ┌─────────┐┌─────────┐┌─────────┐     │
│  │ enqueue ││ allocate││ preempt │     │
│  └─────────┘└─────────┘└─────────┘     │
├─────────────────────────────────────────┤
│          Plugin 层（策略）               │
│  ┌───────┐┌───────┐┌───────┐┌───────┐  │
│  │  gang ││  drf  ││binpack││  ...  │  │
│  └───────┘└───────┘└───────┘└───────┘  │
├─────────────────────────────────────────┤
│          Framework 层（框架）            │
│  Session、Statement、Cache 等基础设施   │
└─────────────────────────────────────────┘
```

**Action（动作）：**
- 调度的 **执行阶段**
- 比如 "enqueue" 是把待调度任务加入队列
- "allocate" 是为任务分配节点

**Plugin（插件）：**
- 调度的 **决策策略**
- 比如 "gang" 决定一组任务要一起调度
- "binpack" 决定尽量把任务堆在同一台机器上

**Framework（框架）：**
- 调度的 **基础设施**
- 提供 Session（调度会话）、Statement（操作记录）等机制

### 3.3 状态存储（ClusterInfo）

**内存中的数据结构：**

```go
type ClusterInfo struct {
    Nodes          map[string]*NodeInfo      // 所有节点
    Jobs           map[JobID]*JobInfo        // 所有任务
    Queues         map[QueueID]*QueueInfo    // 所有队列
    NamespaceInfo  map[NamespaceName]*NamespaceInfo  // 命名空间信息
    RevocableNodes map[string]*NodeInfo      // 可撤销资源节点
}
```

**为什么全放内存？**
- 仿真系统不需要持久化
- 重启即重置，符合实验需求
- 性能好，无数据库依赖

## 四、数据流全景

### 4.1 一次完整仿真的数据流动

```
阶段 1：配置加载
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
YAML 文件 → input_config_loader → Python 字典 → JSON → Go 结构体


阶段 2：初始化
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Python: POST /reset {nodes, workload}
   ↓
Go: 解析 nodes → ClusterInfo.Nodes
Go: 解析 workload → jobQueue (按时间排序)
Go: 初始化 Queues、Namespaces
   ↓
Go: 返回确认 → Python


阶段 3：调度循环
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Python: POST /step {scheduler_conf}
   ↓
Go: 解析调度配置 → acts, tiers, cfg
Go: loadNewSchedulerConf = true (触发调度)
   ↓
Go 主循环：
   • 提交到时间的 Job
   • 推进容器创建
   • OpenSessionV2() → 执行 actions → CloseSession()
   • syncPodPhases()
   • NowTime++
   ↓
Python: GET /stepResult
   ↓
Go: 如果调度完成 → 返回 JSON
Go: 如果还在调度 → 返回 "0"
   ↓
Python: 如果 "0" → 等待后重试
Python: 如果 JSON → 解析结果


阶段 4：报表生成
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
JSON 结果 → flexnpu_util_report → 利用率分析
         → output_csv_reports → CSV 文件
         → SimRun.py → 控制台输出
```

### 4.2 关键数据结构转换

```
用户写的 cluster.yaml
       ↓
input_config_loader.cluster_input_to_simulator_yaml()
       ↓
仿真器格式的 YAML 字符串
       ↓
JSON 传输 (HTTP)
       ↓
Go: simulator.Yaml2Nodes()
       ↓
[]v1.Node → map[string]*NodeInfo
       ↓
存储在 ClusterInfo.Nodes
```

## 五、扩展性设计

### 5.1 如何添加新的调度插件？

**Go 端：**
1. 在 `pkg/scheduler/plugins/` 下新建目录
2. 实现 Plugin 接口
3. 在 `plugins/factory.go` 注册

**Python 端：**
- 无需修改，通过 YAML 配置即可启用

### 5.2 如何添加新的报表类型？

**Python 端：**
1. 在 `output_csv_reports.py` 添加新的 writer 函数
2. 在 `SimRun.py` 调用该函数

**Go 端：**
- 无需修改，只需在 stepResult 中提供必要的数据

### 5.3 如何修改资源模型？

**两边都要改：**
- Go 端：修改资源解析逻辑
- Python 端：修改报表统计逻辑
- 接口：确保双方对资源字段的理解一致

## 六、架构优劣分析

### 优点

| 优点 | 说明 |
|------|------|
| **轻量级** | 单机运行，无需 K8s 集群 |
| **真实** | 运行真实的 Volcano 代码 |
| **可观测** | 详细的报表和日志 |
| **可配置** | YAML 驱动，无需改代码 |
| **可扩展** | 模块化设计，易于扩展 |

### 局限性

| 局限 | 说明 | 可能的改进 |
|------|------|-----------|
| **单节点** | 仿真器只能单机运行 | 可扩展为分布式仿真 |
| **离散时间** | 1 秒粒度，无法模拟亚秒级行为 | 支持更细粒度 |
| **无真实执行** | 任务只是状态变化，没有真正运行 | 可对接真实执行器 |
| **内存限制** | 大规模仿真受限于单机内存 | 数据分片、持久化 |

## 七、总结

VolcanoSimulator 的架构可以概括为：

```
Python 客户端（控制面）
    ↓ HTTP
Go 仿真器（调度面）
    ↓ 调用
Volcano 框架（真实调度代码）
    ↓ 操作
内存状态（虚拟集群）
```

**这种分层、解耦的设计使得：**
1. 你可以只改 Python 端来测试不同的实验场景
2. 你可以只改 Go 端来测试不同的调度策略
3. 你可以独立开发、独立测试、独立部署两边
