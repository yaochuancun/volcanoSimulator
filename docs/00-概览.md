# VolcanoSimulator 通俗详解

> 一句话概括：**这是一个在本地电脑上模拟 Kubernetes + Volcano 调度行为的仿真系统，让你不用搭建真实的 K8s 集群就能测试和验证调度策略。**

---

## 一、这是什么？能解决什么问题？

### 现实世界的痛点

想象你在管理一个 AI 训练集群：
- 有 100 台机器，每台有 8 张 GPU 卡
- 有 1000 个训练任务等着跑
- 任务有大有小，有的需要 8 卡，有的只需要 1 卡
- 你想测试一种新的调度策略，看看能不能让任务完成更快、GPU 利用率更高

**问题来了：**
- 在真实集群上测试？风险太高，可能影响线上业务
- 搭建一套测试集群？成本太高，100 台机器呢！
- 写个简单的程序模拟？太粗糙，无法反映真实调度器的复杂行为

### VolcanoSimulator 的解决方案

这个系统让你在 **一台笔记本电脑** 上就能：
1. **模拟** 出 100 台机器、800 张 GPU 的集群
2. **模拟** 出 1000 个训练任务的提交和执行
3. **运行** 真实的 Volcano 调度器代码（不是简化版！）
4. **观察** 调度结果：哪些任务分配到哪些机器、GPU 利用率如何、任务完成时间多少
5. **测试** 不同的调度策略，对比效果

**核心特点：轻量级、真实、可观测**

## 二、系统长什么样？

### 两大部分，分工明确

```
┌─────────────────────────────────────────────────────────────────┐
│                      VolcanoSimulator                           │
├──────────────────────────────┬──────────────────────────────────┤
│     Python 客户端            │        Go 仿真器                 │
│   (配置 + 控制 + 报表)        │   (调度内核 + 状态模拟)          │
├──────────────────────────────┼──────────────────────────────────┤
│  • 读取 YAML 配置文件         │  • 维护虚拟集群状态              │
│  • 发送 HTTP 请求控制仿真     │  • 运行真实 Volcano 调度代码     │
│  • 接收结果生成报表           │  • 模拟时间流逝和任务执行        │
│  • 分析 FlexNPU 资源利用率    │  • 返回集群快照                  │
└──────────────────────────────┴──────────────────────────────────┘
                    ↑↓ HTTP 通信 (端口 8006)
```

### 形象比喻

想象你在玩 **模拟城市** 游戏：

| 现实世界 | 模拟城市游戏 | VolcanoSimulator |
|---------|-------------|------------------|
| 真实的城市 | 游戏里的城市 | 虚拟的 K8s 集群 |
| 真实的市民 | 游戏里的 NPC | 虚拟的计算任务 |
| 城市管理系统 | 游戏引擎 | Go 仿真器 |
| 游戏存档/读档 | 存档文件 | YAML 配置文件 |
| 游戏界面和统计 | 游戏 UI | Python 客户端 |

**Python 客户端** = 游戏界面 + 存档管理器
- 你选择要加载哪个存档（读 YAML 配置）
- 你点击"开始游戏"（发送 /reset 请求）
- 你点击"下一回合"（发送 /step 请求）
- 你看统计报表（生成 CSV 和利用率报告）

**Go 仿真器** = 游戏引擎
- 维护游戏世界的状态（集群状态）
- 运行游戏逻辑（调度算法）
- 推进游戏时间（仿真时钟）

## 三、工作流程：一次完整的仿真

### 三步走：准备 → 运行 → 观察

```
第 1 步：准备（写配置）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
你写 3 个 YAML 文件：

  cluster.yaml      →  我的"虚拟数据中心"有什么机器？
  workload.yaml     →  我要跑什么任务？
  plugins.yaml      →  用什么调度策略？

第 2 步：运行（Python 控制 Go）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Python 客户端做这些事：

  ① 读 YAML 文件 → 转成 JSON
  ② HTTP POST /reset → "Go 仿真器，初始化集群和任务"
  ③ HTTP POST /step  → "执行一轮调度"
  ④ HTTP GET /stepResult → "现在状态如何？"
  ⑤ 重复 ③④ 直到所有任务都调度完

第 3 步：观察（生成报告）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Python 客户端根据结果生成：

  tasksSUM.csv          →  每个任务的状态和分配位置
  flexnpu_utilization.txt →  GPU 利用率详细分析
  Node_desc.csv         →  每台机器的资源使用
  POD_desc.csv          →  每个 Pod 的详细信息
  npu_chip.csv          →  每张 GPU 卡的利用率
  summary.csv           →  整体统计汇总
```

## 四、核心概念：理解这些就能看懂代码

### 4.1 集群（Cluster）

**现实中：** 一群物理服务器，通过网络连接

**仿真中：** 一个内存里的数据结构，记录每台"虚拟机器"的配置和状态

```yaml
# cluster.yaml 示例
nodes:
  - name: node-1           # 机器名字
    labels:                # 标签（用于调度筛选）
      accelerator: npu
    annotations:           # 注解（存额外信息）
      # 这台机器有 4 张 GPU 卡，每张 100% 算力、8GB 显存
      volcano.sh/flexnpu-core.percentage-list: "[100,100,100,100]"
      volcano.sh/flexnpu-memory.128mi-list: "[64,64,64,64]"
    status:
      allocatable:         # 可用资源
        cpu: "32"
        memory: "128Gi"
```

### 4.2 任务（Job / Workload）

**现实中：** 一个 AI 训练作业，比如用 PyTorch 训练一个模型

**仿真中：** 一个描述文件，说明这个任务需要多少资源、什么时候提交

```yaml
# workload.yaml 示例
jobs:
  - metadata:
      name: training-job-1
      labels:
        sub-time: "0"      # 第 0 秒提交
    spec:
      npuGranularityPercent: 10  # 资源粒度：10% 一档
      tasks:
        - replicas: 4      # 启动 4 个 Pod
          template:
            spec:
              containers:
                - name: train
                  resources:
                    requests:
                      volcano.sh/flexnpu-core.percentage: "35"   # 需要 35% 算力
                      volcano.sh/flexnpu-memory.128mi: "16"      # 需要 2GB 显存
```

### 4.3 调度（Scheduling）

**现实中：** K8s 调度器决定把 Pod 放到哪个节点上运行

**仿真中：** Volcano 调度框架根据配置的策略，决定把虚拟任务分配到哪个虚拟节点

```yaml
# plugins.yaml 示例
scheduler:
  actions: "enqueue, allocate, backfill"  # 调度动作顺序
  tiers:
    - plugins:
        - name: gang        # Gang 调度：确保一组任务同时启动
        - name: drf         # DRF：公平分配资源
        - name: binpack     # Binpack：紧凑打包，提高利用率
```

### 4.4 FlexNPU（GPU 资源模型）

这是这个系统特有的概念，用于模拟 **可切分的 GPU**。

**传统 GPU：** 一张卡要么全给一个人用，要么不给（0 或 1）

**FlexNPU：** 一张 100% 算力的卡可以分成多份给多人用
- 任务 A 用 30%，任务 B 用 50%，剩下 20% 空闲

**为什么要这样？**
- 现实中的 GPU 虚拟化技术（如 MIG、vGPU）支持这种切分
- 仿真系统需要能模拟这种场景

## 六、快速开始：5 分钟上手

### 1. 启动 Go 仿真器

```bash
cd Volcano_simulator/cmd/sim
go build -o sim .
./sim
# 现在它在监听 localhost:8006
```

### 2. 运行 Python 客户端

```bash
cd Submit_volcano_workloads
pip install requests pyyaml prettytable
python SimRun.py
```

### 3. 查看结果

```bash
ls result/2026-04-07-12-00-00/
# 你会看到 tasksSUM.csv、flexnpu_utilization.txt 等文件
```
